## 先分清图里两种“等待”

你这张 **1+2+…+10** 的时空图里，中间其实有两段不同含义：

| 区域 | 周期（约） | 图上长什么样 | 含义 |
|------|------------|--------------|------|
| **空档** | 11～22 | 流水线格子**全空**，没有指令在走 | CPU **暂时不取新指令**，在等前面结果算完 |
| **斜向阴影** | 23～30 | 阴影块沿 取→译→执→访→写 **斜着往下移** | 一条 **bubble（NOP）** 在各级里 **被“冲”过去** |

右边手写：`AX=(1+2)+5+7+9`，`CX=(3+4)+6+8+10`，最后再 **`AX+CX`**。  
所以中间等待的目的很简单：**最后一条加法必须等 AX、CX 都准备好**，不能提前做。

## 是不是 Hazard Unit 的 stall？

**不完全是，而且通常也不是你现在课设里那种 load-use stall。**

| 机制 | 典型长度 | 触发原因 | 和你这张图的关系 |
|------|----------|----------|------------------|
| **Hazard Unit（load-use）** | **1 拍** | `LW` 后一条立刻用加载结果 | 斐波那契/五级流水课设里的标准 stall |
| **Hazard Unit（分支 flush）** | **约 2 拍** | `BNE` 跳错路 | 和 1～10 求和无关 |
| **图中间 11～22 的空档** | **很多拍** | 算法/控制器 **故意不发起** 下一条 | **不是** load-use 那种 1 拍 stall |
| **图 23～30 的阴影** | **约 1 条 NOP 穿过五级** | 插入 bubble | **看起来像** stall 的视觉效果，但可能是 **微程序等待、NOP、或同步**，不一定是 Hazard Unit |

结论：

- **长等待（11～22）**：主要是 **“等数据就绪 / 等微程序做完”**，不是 Hazard 检测 RAW 自动插 1 拍那种。
- **斜阴影（23～30）**：才是 **bubble 在流水线里传播** 的画法；实现上 **可以** 用 stall，也 **可以** 用别的方式。

## 这种“等待”实际怎么实现？（三种常见做法）

### 1. 多周期模型机 / 单总线（更可能对应这张图）

老模型机用 **AX/BX/CX/DX + 微命令**，没有五级流水 Hazard Unit，等待靠：

- 微程序里 **多排几个空微步**（等价 NOP）
- 或遵守 **「本周期决策，下周期生效」**：例如 `MOVE AC,[BX]` 之后，**下一微周期才能用 AC**，时空图上就会自然空出格子

```text
不是 Hazard Unit，而是 CM_ROM 里写死的“等一拍”
```

### 2. 五级流水 + 程序里插 NOP（软件等待）

```asm
; 前面算完 AX、CX 之后
NOP
NOP
...
ADD  final, AX, CX   ; 等够再算最后一步
```

汇编多写几条 NOP，时空图上就会出现 **空档 + 斜着移动的 bubble**。

### 3. 五级流水 + Hazard Unit stall（硬件，但一般是短 stall）

只有检测到 **load-use、结构冲突、Cache miss** 等才：

```text
Pc_en = 0, Ifid_en = 0, Control_src = 0  → 插 1 个 bubble
```

**不会**为了 “等 AX、CX 都算完” 自动 stall 十几个周期——那是 **程序依赖**，应靠 **转发** 或 **等前序指令自然流过流水**，而不是 Hazard 专门“等 1+10”。

对 **1+2+…+10**，若写成流水 RISC、且有 **forward**，中间 **不必** 画 12 拍空档；最后 `ADD AX,CX` 只要隔够 1～2 条或靠转发即可。

## 斜阴影块 = bubble 在动（若用 stall，就是这样）

若 **确实** 用 Hazard Unit（或等价逻辑）插了 1 个 bubble，五级流水里会看到：

```text
周期 N:   往 ID/EX 灌入全 0 控制（NOP）
周期 N+1: NOP 在 EX
周期 N+2: NOP 在 MEM
...
```

图上 **阴影斜着扫过 取、译、执、访、写**，就是这个 NOP **逐级往下漂** 的标准画法。

对应信号（你前面问的 Hazard Unit）：

```text
Pc_en = 0        → 不取新指令
Ifid_en = 0      → IF/ID 冻住
Control_src = 0  → ID/EX 进 NOP（bubble）
```

## 和斐波那契课设的关系

| 项目 | 1+10 这张图 | 你的五级流水斐波那契 |
|------|-------------|------------------------|
| 寄存器 | AX/CX… | x1～x5 |
| 中间长等待 | 等两半和都算完再做 `AX+CX` | 循环里用 **forward** 即可 |
| Hazard Unit | 多半 **不是** 主角 | **load-use stall + 分支 flush** |

斐波那契里 **没有** 这种“中间空十几拍再等”的设计；RAW 用 **forward**，只有 `LD` 紧挨使用才 **stall 1 拍**。

---

## 一句话

- **中间长等待（11～22）**：为实现 **最后 `AX+CX` 前两边结果都就绪**；在多周期模型机里常是 **微程序/总线时序等待**，不是 Hazard Unit 的典型 load-use stall。  
- **后面斜阴影（23～30）**：表示 **一条 bubble/NOP 穿过流水线**；若用硬件做，机制 **类似** stall（`Pc_en/Ifid_en/Control_src`），但 **1+10 整张图的长空档不能简单说成“全靠 Hazard Unit stall”**。  

若老师要求“用 Hazard Unit 实现”，应明确是 **哪一种 hazard**（load-use / 结构 / Cache ready），而不是“等为 1～10 算完再算最后一步”——后者是 **数据依赖 + 程序顺序**，不是互锁检测那一类。
