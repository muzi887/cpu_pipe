# Forward MUX A/B 和 ALUSrc MUX 

## 1. 整体结构（先看清顺序）

ALU 有两个输入 **A** 和 **B**，来源不同：

```text
                    ┌──────────────── Forward MUX A ────────────────┐
                    │  00: ID/EX.RS                                  │
                    │  01: EX/MEM.alu_result                         ├──> ALU(A)
                    │  10: MEM/WB.write_data                         │
                    └────────────────────────────────────────────────┘

                    ┌──────────────── Forward MUX B ────────────────┐
                    │  00: ID/EX.RD                                  │
                    │  01: EX/MEM.alu_result                         │
                    │  10: MEM/WB.write_data                         │
                    └───────────────┬────────────────────────────────┘
                                    │
                                    v
                              ┌───────────┐
                              │ ALUSrc MUX│
                              │  0: 上一路 │
                              │  1: imm   │──> ALU(B)
                              └─────▲─────┘
                                    │
                              ID/EX.imm
```


---

## 2. 三个 MUX 各自干什么

### Forward MUX A / B（转发多路选择器）

**作用：** 给 ALU 的 A、B **选“操作数从哪来”**——正常从 ID 读的寄存器，还是从流水线后面**转发**来的结果。

| Forward 值 | 选什么 |
|------------|--------|
| 00 | ID/EX 里 ID 阶段读出的 RS（A）或 RD（B） |
| 01 | 上一条在 EX/MEM 的 **alu_result** |
| 10 | 上一条在 MEM/WB 的 **write_data** |

**为什么需要两个（A 和 B）？**

ADD 两个源操作数可能都有相关，例如：

```asm
ADD  x3, x1, x2
ADDI2 x4, x3, 1   ; 同时需要 x3 和立即数
```

- A 可能需要转发 **x3**
- B 可能是 **imm**，也可能也需要转发  

所以 **A、B 各一个 Forward MUX**，互不干扰。

---

### ALUSrc MUX（只在 B 侧）

**作用：** 在「寄存器/转发后的寄存器值」和 **立即数 imm** 之间二选一，**只接在 B 路上**。

| ALUSrc | B 选什么 |
|--------|----------|
| 0 | Forward MUX B 的输出（寄存器操作数） |
| 1 | ID/EX.imm（立即数） |

**为什么只在 B 侧？**

常见约定：

- **A** = 第一个寄存器（rs1）
- **B** = 第二个寄存器（rs2）**或** 立即数（imm）

Control 用 **ALUSrc** 区分 ADD 和 ADDI，不需要在 A 侧再选 imm。

---

## 3. 两条路径对比

```text
ALU 输入 A 的路径（简单）：

  ID/EX.RS ──> Forward MUX A ──> ALU(A)
              （解决 RAW 转发）


ALU 输入 B 的路径（多一级）：

  ID/EX.RD ──> Forward MUX B ──> ALUSrc MUX ──> ALU(B)
              （解决 RAW 转发）    （选 reg 或 imm）
                                    ↑
                              ID/EX.imm
```

**B 路多 ALUSrc**，是因为 B 既可能是 **rs2**，也可能是 **imm**；  
**A 路**一般始终是寄存器（或 x0），所以通常 **没有 ALUSrc A**。

---

## 4. 用两条指令举例

### 例 1：`ADD x3, x1, x2`

```text
Control: ALUSrc = 0, ForwardA = 00, ForwardB = 00

ALU(A) = RS = x1
ALU(B) = RD = x2        （Forward B 输出 → ALUSrc 选 0 → 不用 imm）
result = x1 + x2
```

### 例 2：`ADDI2 x2, x3, 0`（紧接在 ADD x3,x1,x2 后面）

```text
Control: ALUSrc = 1, ForwardA = 01

上一条 ADD 的 result 还在 EX/MEM：
  Forward MUX A 选 01 → ALU(A) = 上一条的 alu_result（新 x3）
  ALUSrc = 1 → ALU(B) = imm = 0

result = x3 + 0 = x3  →  写回 x2
```

这里：

- **Forward MUX A** 解决「x3 还没写回就要用」
- **ALUSrc MUX** 选 imm=0，不选 RD

### 例 3：`ADD x3, x1, x2` 且 x2 也需要转发

```text
ForwardA = 00  → A = x1
ForwardB = 10  → B = 从 MEM/WB 转发的 x2 新值
ALUSrc = 0     → 不用 imm
```

说明 **A、B 的 Forward 可以独立设置**。

---

## 5. 为什么 B 的顺序必须是 Forward → ALUSrc

顺序不能反：

```text
错误：ALUSrc 在前，Forward 在后
  → imm 也会被 Forward 逻辑干扰，控制变复杂
```

正确顺序：

```text
1. Forward MUX B：先确定「寄存器操作数」是否正确（含转发）
2. ALUSrc MUX：再决定用这个寄存器值，还是换成 imm
```

ADD 时：Forward B 给出正确的 rs2（可能转发）→ ALUSrc=0 → 进 ALU  
ADDI 时：Forward B 的输出可忽略 → ALUSrc=1 → 直接用 imm 进 ALU

---

## 6. 一句话记忆

| 部件 | 位置 | 解决什么问题 |
|------|------|--------------|
| **Forward MUX A** | ALU(A) 前 | A 操作数的 RAW 转发 |
| **Forward MUX B** | ALU(B) 前 | B 操作数（寄存器侧）的 RAW 转发 |
| **ALUSrc MUX** | Forward B 之后 | B 用寄存器还是用 **imm** |

**ALUSrc MUX 前面确实有 Forward MUX B**；A 侧只有 Forward MUX A，没有 ALUSrc。整条 B 路是：

```text
Forward MUX B → ALUSrc MUX → ALU(B)
```
