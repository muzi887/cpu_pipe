##  转发

转发（Forwarding / Bypassing）：不等到 WB 写回寄存器，把中间结果直接从产生处送到 EX 的 ALU 输入。

正常路径：EX → MEM → WB → Registers →（下一指令 ID 再读）

转发路径：EX/MEM 或 MEM/WB ──直接──> 下一指令 EX 的 ALU

图 2 示意：

| 依赖指令        | ① 产生 R2 的位置   | 需要 R2 的指令    | 转发来源                        |
| ----------- | ------------- | ------------ | --------------------------- |
| ② SUB R3,R2 | ① 在周期 3 的 EX  | ② 在周期 4 的 EX | EX/MEM.alu_result → ② 的 ALU |
| ③ AND R4,R2 | ① 在周期 4 的 MEM | ③ 在周期 5 的 EX | MEM/WB 结果 → ③ 的 ALU         |
| ④ ADD R5,R2 | ① 在周期 5 的 WB  | ④ 在周期 5 的 ID | 不转发，靠寄存器堆先写后读               |

#### ④ 要用 R2 时
① 的结果刚好在同一拍写回寄存器堆；不靠 EX 前的 Forward MUX 旁路，而是让 ID 读端口在同一时钟周期内读到 WB 刚写进去的新值。

若 WB 写和 ID 读落在同一时钟周期、同一寄存器（这里都是 R2），就要约定谁先谁后，否则 ID 可能读到写之前的旧值。

先写后读 = 在一个时钟周期内约定：

1. 先：WB 把 ① 的结果写入 R2（写口生效）
2. 后：ID 用 `rs` 读 R2（读口看到的是刚写入的新值

常见实现方式包括：

- **上升沿写、下降沿读（或前半拍写、后半拍读）**
- 写口与读口地址相同时，寄存器堆内部的 write-to-read bypass（读口直接看到本拍要写进去的数据）

因此 ④ 在周期 5 的 ID 读到的就是 ① 刚算出的 R2，不必 stall，也不必在 EX 加转发 MUX。


> [!NOTE]  ① 在周期 5 的 WB 写入 R2 与 ④ 在周期 5 的 ID 读取 R2 发生在同一拍；寄存器堆设计成本周期内先完成写、再完成读（或内部旁路）


#### 硬件上就是 Forward MUX：

ForwardA/B = 00 → 用 ID/EX 里读出的 RS/RD（正常）

ForwardA/B = 01 → 用 EX/MEM.alu_result

ForwardA/B = 10 → 用 MEM/WB.write_data

检测条件（简化）：

若 EX/MEM.Reg_w=1 且 EX/MEM.rd = 当前需要的 rs

→ 从 EX/MEM 转发

否则若 MEM/WB.Reg_w=1 且 MEM/WB.rd = 当前需要的 rs

→ 从 MEM/WB 转发

### stall
Stall（停顿 / 气泡 / 流水线冻结） 指：流水线遇到无法立刻解决的冲突时，故意空转 1 个或多个时钟周期，让前面指令多走几步，等数据或控制结果就绪后再继续。

可以记成：该前进的地方不前进，该进下一级的插入“空指令（NOP/bubble）”。

典型 load-use stall（最常见）：

LD x3, 0(x4) ; 从内存读 x3

ADD x5, x3, x1 ; 下一条立刻用 x3

时序问题：

LD: IF ID EX MEM WB

ADD: IF ID EX MEM WB

↑

这里要用 x3，但 LD 的数据 MEM 才出来

Hazard Unit 检测到 load-use 后：

|动作|含义|
|---|---|
|PC_hold = 1|PC 不变，不再取新指令|
|IF/ID_hold = 1|IF/ID 寄存器不变，重复译码同一条|
|ID/EX_flush = 1|向 ID/EX 插入 bubble（NOP），相当于空泡|

效果示意：

正常：

周期3: LD在EX, ADD在ID

周期4: LD在MEM, ADD在EX ← ADD要用x3，但还没有！

Stall 1拍后：

周期3: LD在EX, ADD在ID

周期4: LD在MEM, ADD在ID ← ADD停在ID，不进入EX（插入bubble）

周期5: LD在WB, ADD在EX ← 此时可转发或读到x3
