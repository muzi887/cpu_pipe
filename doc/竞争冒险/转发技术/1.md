# 第 1 步改动的详细解释：为什么这样改、每部分是什么意思

你当前图是「ID 读出 RS/RD → 一路传到 WB → MUX 选一个写回」。这只能**传数据**，不能**算数**，且 **Wr_addr 接错** 会在流水里写错寄存器。下面按块说明第 1 步要改什么、为什么。

---

## 一、整体目标：从「传寄存器值」变成「算完再写回」

斐波那契至少需要：

```asm
ADDI x1, 1        ; x1 ← 0 + 1
ADD  x3, x1, x2   ; x3 ← x1 + x2
ADDI2 x1, x2, 0   ; x1 ← x2 + 0
```

这些指令的语义都是：

```text
rd ← rs1 OP rs2/imm
```

不是「把 RD 原样写回去」。所以 EX 必须有 **ALU**，WB 必须写 **ALU 结果**，不能写 RS/RD 原值。

---

## 二、EX 阶段：ALU 和两个 MUX
ALU 的 B 输入前面有两级选择——先 Forward MUX B，再 ALUSrc MUX。 A 输入前面通常只有 Forward MUX A。

**ALU.result** 送入 **EX/MEM.alu_result**，再经 MEM/WB 到 WB 写回。

---

### 2. ALUSrc MUX：ALU 第二输入选谁

```text
Forward MUX B ──> ALUSrc MUX ──> ALU(B)
                      ↑
                 ID/EX.imm
                 
ALUSrc = 0 → 选寄存器（RD / rs2）
ALUSrc = 1 → 选立即数 imm
```

**含义：**

- **ADD**：两个操作数都来自寄存器 → `ALUSrc=0`，B = RD（或 rs2）
- **ADDI / ADDI2**：一个是寄存器、一个是立即数 → `ALUSrc=1`，B = imm

**原因：** 16 位指令里 imm 在 `[3:0]` 或 opcode 低位，**不在寄存器堆**。ID 阶段用扩展器变成 16 位 imm，锁进 ID/EX，EX 再送给 ALU。

没有 ALUSrc，ADDI 无法把 imm 送进 ALU，只能做「两个寄存器相加」。

**数据流示意：**

```text
ADD  x3, x1, x2:
  ALU(A) = RS = x1
  ALU(B) = RD = x2     (ALUSrc=0)

ADDI x1, 1:
  ALU(A) = 0 (x0)
  ALU(B) = imm = 1     (ALUSrc=1)
```

---

### 3. Forward MUX A / B：放在 ALU 前面

```text
Forward MUX A ──> ALU(A)
Forward MUX B ──> ALUSrc MUX ──> ALU(B)
```

**含义：** 为 ALU 的 A、B **选数据来源**，不只来自 ID/EX 里 ID 阶段读出的 RS/RD。

可选来源（典型）：

| Forward | 来源 | 何时用 |
|---------|------|--------|
| 00 | ID/EX.rs1_val / rs2_val | 正常，无相关 |
| 01 | EX/MEM.alu_result | 上一条在 MEM，结果刚算完 |
| 10 | MEM/WB.write_data | 上一条在 WB，即将/正在写回 |

**原因：流水线重叠导致「读太早」**

```asm
ADD  x3, x1, x2    ; 周期 N 在 EX 才算出 x3
ADDI2 x2, x3, 0    ; 周期 N+1 在 EX 就要用 x3
```

第二条在 EX 要用 x3 时，第一条可能还在 **EX/MEM** 或 **MEM/WB**，**尚未 WB 写进 Registers**。若只从寄存器堆读，会读到**旧 x2**。

Forward MUX 把 **EX/MEM.alu_result** 或 **MEM/WB.write_data** 直接送到 ALU，不等 WB。

**为什么叫「Forward」：** 把后面阶段的结果**提前（forward）** 送到前面 EX 的 ALU。

第 1 步若只做 ADDI/ADD、且指令间隔足够，可先不加 Forward，但斐波那契循环里 **ADD → ADDI2 x2,x3,0** 紧挨着，**必须加**，否则 x2 会错。



```text
        ┌─ ① EX/MEM.alu_result     （转发，上）
        │
ID/EX ──┼─ ② ID/EX.RS              （正常读寄存器，中间）
.RS     │
        └─ ③ MEM/WB.write_data     （转发，下）
                │
                v
              MUXA ──> ALU(A)
```



---

### 4. ALU.result → EX/MEM.alu_result

```text
ALU.result ──> EX/MEM.alu_result
```

**含义：** EX 组合逻辑算出的结果，在时钟沿锁进 **EX/MEM**，供 MEM、WB 及转发使用。

**原因：**

1. **段间隔离**：EX 是组合逻辑，MEM/WB 是后续拍，必须用流水线寄存器隔开。
2. **转发源**：Forward MUX 的 01 选 **EX/MEM.alu_result**。
3. **写回源**：最终 WB 的 writedata 来自 **MEM/WB.alu_result**（经 MemToReg，第 1 步只有 ALU 结果）。

---

## 三、WB 修改：删掉旧 MUX，直接写 ALU 结果

### 旧图问题

```text
MEM/WB.RD ──┐
            ├── MUX(Rd_src) ──> writedata
   Rd_src ──┘
```

这是在 **RD 寄存器原值** 和 **Rd_src** 之间选，相当于：

```text
rd ← rd   或   rd ← 某个外部源
```

**不能**实现 `rd ← rs + imm`，也不能实现 `rd ← rs1 + rs2`。

### 新写法

```text
MEM/WB.alu_result ──> writedata ──> Registers
Reg_w = 1 时：Registers(Wr_addr) ← writedata
```

**含义：** 写回的是 **运算结果**，不是读出的 RS/RD。

**原因：** 算术/立即数指令的语义就是「把 ALU 输出写进 rd」。第 1 步只有 ADD/ADDI，没有 Load，故暂不需要 MemToReg；以后加 LD 再在 WB 前加 **alu_result vs mem_data** 的 MUX。

---

## 四、Wr_addr 必须流水线传递：为什么不能直接用 Ins[7:4]

### 错误接法

```text
WB 阶段：Wr_addr ← 当前 IF/ID 的 Ins[7:4]
```

### 为什么错：流水里同时有多条指令

周期 5 示例：

```text
周期 5 各阶段同时在跑：
  WB：指令① ADD x3,x1,x2  应写 x3
  MEM：指令② ST ...
  EX：  指令③ ADDI2 x1,x2,0
  ID：  指令④ ADDI2 x2,x3,0   ← Ins[7:4] 可能是 x2
  IF：  指令⑤ BNEZ ...
```

若 WB 用 **当前 ID 正在译码的 Ins[7:4]**：

- ① 要写 **x3**
- ID 里可能是 **x2**
- 结果：**x2 被写成 x3 的值** → 完全错误

### 正确接法

```text
ID：Ins[7:4] 作为 rd
    ↓ 锁存
ID/EX.rd
    ↓
EX/MEM.rd
    ↓
MEM/WB.rd
    ↓
WB：Wr_addr ← MEM/WB.rd
```

**含义：** rd 在 **ID 译码那条指令时** 确定，之后 **随该指令在流水线里一起走**，WB 时用 **这条指令自己的 rd**，不受 IF/ID 里新指令影响。

**原因：** 流水线 = 多条指令各阶段并行；**地址、数据、控制信号都要和指令绑定**，经流水线寄存器传递。这叫 **「控制/数据与指令同行」**。

---

## 五、ID/EX 要锁存什么：`{ RS, RD, imm, rd, control }`

```text
ID/EX ← { RS, RD, imm, rd, control(EX段) }
```

| 字段 | 来源 | EX 用途 | 为何要锁存 |
|------|------|---------|------------|
| **RS** | Registers 读口 | ALU(A) 或 Forward A | ID 是组合读，EX 要用上一拍 ID 的结果 |
| **RD** | Registers 读口 | ALU(B) 或 Forward B | 同上 |
| **imm** | 扩展器 | ALU(B) when ALUSrc=1 | imm 来自 ID 那拍的指令 |
| **rd** | Ins[7:4] | 传到 WB 作 Wr_addr | 见上一节 |
| **control** | Control 单元 | ALUSrc, ALUOp, Reg_w 等 | EX 按「这条指令」的控制信号工作 |

**原因：**

1. **时序**：ID 组合逻辑输出在时钟沿前要稳定；上升沿锁进 ID/EX，EX 整拍使用。
2. **隔离**：EX 工作时 IF/ID 已是下一条指令，不能再用「当前 Ins」读 RS/RD。
3. **控制传递**：Reg_w 等要在 WB 才用，需 ID/EX → EX/MEM → MEM/WB 一路传。

**control(EX段)** 至少含：`ALUSrc, ALUOp`；完整版还有 `RegWrite, MemRead, MemWrite, MemToReg, Branch` 等，随流水向后传。

---

## 六、用一条 ADD 走一遍（串起来）

**`ADD x3, x1, x2`**（x3 ← x1 + x2）

| 阶段 | 发生的事 |
|------|----------|
| **IF** | 取指，Inst 进 IF/ID |
| **ID** | RS_addr=1→RS=x1；RD_addr=2→RD=x2；rd=3 锁入 ID/EX；Control：ALUSrc=0, ALUOp=ADD, Reg_w=1 |
| **EX** | Forward 选 RS/RD（或转发）；ALU：x1+x2=result；result→EX/MEM；rd=3 一起传 |
| **MEM** | 不访存，alu_result 直通 |
| **WB** | writedata=alu_result；Wr_addr=3；Reg_w=1 → **x3 ← x1+x2** |

若下一条是 **`ADDI2 x2, x3, 0`** 且紧挨着：

| 阶段 | 转发 |
|------|------|
| **EX** | 需要 x3，上一条 ADD 的 result 在 EX/MEM → **ForwardA=01**，ALU(A)=alu_result，不用寄存器里旧 x3 |

| 能力 | 状态 |

## 八、和你原图的对应关系

| 原图 | 第 1 步改后 |
|------|-------------|
| ID/EX 只传 RS、RD | 再加 **imm、rd、control** |
| EX 空过 | 加 **Forward MUX + ALUSrc + ALU** |
| EX/MEM 只传 RD | 传 **alu_result、rd、control** |
| WB MUX(RD, Rd_src) | 删；**writedata = alu_result** |

## 九、一句话总结

1. **ALU + ALUSrc**：实现 `rd ← rs OP imm/reg`，而不是原样传 RD。  
2. **Forward MUX**：解决流水重叠时「上一条结果还没写回，下一条就要用」的问题。  
3. **alu_result 写回**：WB 写运算结果，不是寄存器旧值。  
4. **rd 流水线传递**：WB 写哪个寄存器必须跟「正在 WB 的那条指令」绑定，不能用当前 ID 的 Ins[7:4]。  
5. **ID/EX 锁存**：把 ID 拍读出的 RS/RD/imm/rd/control 留给 EX 用，这是五级流水的基本规则。
